# HeteroNetwork PostgreSQL HA

HeteroNetwork stores node identity records, endpoint candidates, health,
service leases, routes, client gateway selections, and the join-token ledger in
PostgreSQL. Database placement is part of the standard Linux enrollment flow;
it does not require a second install command, copied environment variables, or
manual certificate distribution.

## Automatic Placement

Run the install command generated by **Add device** on each Linux server. The
same command installs and starts:

- the HeteroNetwork Agent and dynamic public gateway;
- `heteronetwork-postgres-autopilot.service`;
- the PostgreSQL/Patroni/DCS renderer used by the autopilot.

The autopilot waits until at least three enrolled Linux nodes can reach each
other through HeteroNetwork. The lowest ready database member creates the first
topology. Other nodes download the authenticated bootstrap bundle over the
encrypted overlay and configure themselves without operator input.

Every ready non-client Linux node becomes a synchronous PostgreSQL replica, up
to 32 members. Public Internet reachability is irrelevant to placement. The
first three members form the initial etcd DCS quorum. When at least five
database members exist, the autopilot expands the DCS to five voters one at a
time:

1. add one etcd learner;
2. start it with the existing-cluster configuration;
3. wait until etcd allows promotion;
4. promote it and repeat for the fifth voter.

This avoids changing quorum before a new member has copied the Raft log. A
five-voter DCS requires two synchronous PostgreSQL standbys, so an acknowledged
write is present on the primary and two replicas. The steady-state topology can
lose any two correctly distributed database/DCS members without losing
acknowledged data or DCS quorum.

The autopilot reconciles every 30 seconds. A later Linux enrollment therefore
adds a PostgreSQL replica automatically and, while the DCS has fewer than five
voters, also advances the learner workflow. A temporarily unreachable existing
database member is retained in the topology and rejoins from the current
Patroni timeline when it returns.

## Replicated Bootstrap Material

The database CA, per-member certificates, and generated database credentials
are stored with mode `0600` below:

```text
/etc/heteronetwork/postgres-autopilot/bundle
```

Each database member serves an authenticated copy only on its HeteroNetwork
address. The bearer is derived by the Control Plane from the cluster ID and
node-enrollment signing key, then written to a root-only systemd environment
file. The signing key itself is never included in an enrollment response.

Every member keeps the complete private bundle so loss of the node that created
the cluster does not prevent later enrollment or certificate issuance.
Compromise of root on any database member must therefore be treated as a
database-cluster credential compromise.

Do not expose the following ports to the Internet:

```text
17446  authenticated autopilot bundle replication
55432  PostgreSQL
18008  Patroni REST/TLS health
12379  etcd client TLS
12380  etcd peer TLS
25432  local primary-selecting HAProxy
```

## Installed Services

Database members run:

```text
heteronetwork-postgres-autopilot.service
heteronetwork-postgres-bundle.service
heteronetwork-db.service
heteronetwork-db-proxy.service
```

The current three to five DCS members also run:

```text
heteronetwork-db-dcs.service
```

Use systemd and Patroni status for bounded operational inspection:

```bash
sudo systemctl status \
  heteronetwork-postgres-autopilot.service \
  heteronetwork-db.service \
  heteronetwork-db-proxy.service
sudo journalctl -u heteronetwork-postgres-autopilot.service --since '15 minutes ago'
sudo /opt/heteronetwork/postgres-ha/patroni/bin/patronictl \
  -c /etc/heteronetwork/postgres-ha/patroni.yml list
```

## Failure Test

After at least five DCS voters and five PostgreSQL members are healthy:

1. write a unique value through local port `25432`;
2. stop both database services on the current primary;
3. confirm another member becomes primary and returns the value;
4. stop a second member and repeat the write/read check;
5. restart both members and require them to return as streaming replicas.

The public gateway role and database role are independent. The failure test
must therefore include private-only database members as well as public gateway
hosts.

Synchronous replication is not a backup. Keep encrypted base backups and WAL
archives in an independently administered object store for accidental deletion,
corruption, credential compromise, and loss of all failure domains.

## Recovery Helper

`scripts/postgres-ha-node.sh` remains available for renderer self-tests,
inspection, and controlled recovery. Its `init-bundle`, `extend-bundle`,
`reconcile-dcs`, and `reconfigure-node` commands are internal primitives used
by the autopilot. They are not part of the normal node-installation procedure.
