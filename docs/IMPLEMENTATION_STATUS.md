# Implementation Status

This file tracks the gap between the requested final system and the current repository state.

## Implemented In This Baseline

- Rust workspace with dedicated crates for types, crypto, control plane, signal, relay, STUN, agent, route manager, and CLI.
- Typed models for node, peer/path state, relay capability, token, policy, ACL, route, and health.
- Ed25519 signed join token creation and verification.
- X25519/WireGuard key generation primitive.
- Pair-scoped path scoring with direct/IPv6/NAT traversal/relay/unreachable states.
- In-memory control-plane registration and store-aware VPN IP allocation that skips already assigned addresses after durable-store restarts and retries after durable-store VPN IP insert races.
- SQLite and PostgreSQL control-plane store implementations with SQLite round-trip tests, VPN IP uniqueness indexes, and typed duplicate node-ID/VPN-IP insert errors.
- Token ledger records, control-plane revocation API, revocation state, and race-safe max-use enforcement for in-memory and SQL stores using compare-and-swap SQL token-use updates.
- Control-plane join service that validates signed join tokens, issuer keys, overlapping issuer key rotation sets, cluster/time constraints, ledger admission, CIDR-containing route policy, relay-capability policy, Ed25519 identity-public-key format, WireGuard public-key format, identity-derived node IDs, candidate ownership, endpoint candidate kind/address-family consistency, route ownership, and node registration.
- Control-plane relay maps and relay-candidate metrics require relay capability policy, capacity, E2E-only relay mode, and a fresh healthy heartbeat inside the configured relay health TTL.
- Control-plane heartbeat updates require a fresh node-identity signature, verify against the registered identity public key, reject replayed or older signed heartbeats, store the accepted signature timestamp as health freshness time, and reject candidate/path-state updates that do not belong to the reporting node.
- Control-plane WireGuard key rotation API requires a fresh node-identity signature over the previous and next WireGuard public keys, validates both key formats, rejects stale current-key transitions, persists the updated node record in memory/SQLite/PostgreSQL stores, and returns refreshed peer/relay maps.
- Control-plane HTTP crate with typed health, join, signed WireGuard key rotation, policy inspection, ACL-filtered and candidate-fresh peer-map, and JSON/Prometheus metrics routes backed by the join service.
